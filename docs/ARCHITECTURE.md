# Architecture

## Overview

The dashboard has three data layers:

1. Historical artifacts
2. Shared live snapshot
3. Private live scenarios

Historical artifacts are generated offline and committed to the repository. Live state is read from a shared snapshot by default. Users can also run private live scenarios without mutating shared state.

## Historical Layer

Historical analysis is driven by generated files under `public/data/`.

Key files:

- `meta.json`: asset registry, event registry, tags, macro context, availability
- `event_returns.json.gz`: precomputed historical event windows
- `daily_history.json.gz`: raw validated daily history used for local custom events and live payload generation
- `trigger_zscores.json`: historical trigger normalization inputs
- `last_updated.json`: provenance, schema version, as-of date, and refresh warnings

Load path:

- `src/hooks/useData.ts` loads artifacts on app startup
- labels and event names are normalized on load
- FRED failures and removed assets are surfaced as provenance warnings

## Live Layer

### Shared live snapshot

The shared snapshot is generated by the data pipeline and written to `public/data/live_snapshot.json`.

Default behavior:

- `src/app/api/live-pull/route.ts` returns the shared snapshot directly when the request matches the default shared view
- all normal users read that same snapshot by default

Snapshot fields include:

- `requestedDay0`
- `actualDay0`
- `snapshotDate`
- `asOfDate`
- `dayN`
- `tradingDayN`
- display returns/levels
- scoring returns/levels
- asset status and warnings

### Private live scenarios

When a request is not served by the shared snapshot:

- the route rebuilds a live payload from generated daily history where possible
- only the private response is affected
- shared snapshot state is not mutated

## No-Lookahead Policy

The live engine uses the latest observed market date on or before the requested Day 0. This policy is enforced in the live API and shared helper layer.

Implications:

- weekend and holiday requests anchor to the previous valid trading session
- analogue scoring, gate logic, and trade ideas never use future observations to define Day 0

## Scoring Model

Core scoring logic lives in `src/engine/similarity.ts`.

Inputs:

- live scoring returns
- selected similarity assets
- active event universe
- tag overlap
- macro context

Composite score:

- quant path similarity
- tag similarity
- macro similarity

Weights are normalized to 1 inside the store before scoring.

## Local-Only Custom Events

Custom events are intentionally browser-local.

Rules:

- selected date must be inside loaded historical coverage
- anchor date resolves on or before the selected calendar date
- returns are computed from local daily-history data only
- custom events persist in local storage
- custom events do not modify shared artifacts or team-wide state

Implementation:

- `src/engine/customEvents.ts`
- `src/components/tabs/historical/EventsTab.tsx`
- `src/store/dashboard.ts`

## State Management

Global dashboard state is stored in Zustand under `src/store/dashboard.ts`.

Key state domains:

- historical metadata and artifacts
- live payload and provenance
- analogue scores and weights
- active event selection
- local custom events
- cross-asset selection

## Main UI Surfaces

- Historical: event selection, overlays, heatmaps, cross-asset comparisons, step-in analysis
- Live: live config, analogue scoring, paths, trade ideas, detail
- Analysis: screener, reverse lookup, lead-lag, pre-positioning, rotation
- Risk: decay, stress, confidence, OOS, gate
- Tools: correlation, memo

Most live/risk/tools tabs now share the same effective scoring day logic, so the app does not mix chart-display offsets with scoring offsets.
